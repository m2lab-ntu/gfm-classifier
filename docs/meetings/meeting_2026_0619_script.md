## Slide 1 — Title

老師好。今天 6/19，接續上禮拜 6/9 那次報告後的這一週。

先講一下出發點。上禮拜結束的時候，我們的 NT-v2 genus classifier 在 50M 的
balanced data 上已經做到 66.6%，那個是 v9。所以這禮拜的計畫其實很單純——
就是把 training data 從 50M 放大到完整的 258M，想說資料變 5 倍，accuracy
應該可以再往上推個 2 到 5 個百分點。

結果完全相反。放大到 258M 之後，accuracy 不但沒漲，反而掉到大概 44.5%，
等於少了 22 個百分點。所以這禮拜我大部分的時間，其實不是在訓練，而是在
debug 這件事——為什麼資料越多反而越差。我把根因找出來了，而且修正版現在
已經在跑，結果待會兒會給老師看。

整份大概九張，我講 12 到 15 分鐘。

## Slide 2 — 進度時間軸 / Timeline

這張是時間軸，我從上禮拜 6/9 一路講到今天。

6/11 我先把 v10 丟出去。那時候的設定是：258M 完整資料、從 v9 的 best checkpoint
做 warm-start，然後因為資料很不平衡，我就「想當然耳」開了 class_weights。
現在回頭看，這個決定是錯的，但那是後話。

6/13 結果出來，整個 collapse。100K 的獨立 test set 只有 40.1%，比 v9 還低一大截，
而且訓練過程 val_acc 從頭到尾卡在 21% 完全不動。

6/14 我第一個直覺是：會不會是 class_weights 在 400:1 這種不平衡下，把 loss 搞壞了？
為了驗證，我一次丟了兩個對照組——v12 是 warm-start、v13 是 from scratch，
兩個都把 class_weights 關掉。

6/15 到 6/18，這兩個都收斂在 44.5% 左右。重點是：warm-start 跟 from scratch 幾乎沒差，
weights 關掉也沒救回來。那這就很奇怪了——我換了初始化、換了加權，天花板完全不動。
這種「怎麼調都一樣」的型態，通常代表問題不在這些地方，而在更上游。所以我就回頭去
audit 程式跟資料，然後就挖到兩個 bug。

第一個是，資料集根本用錯了。我餵進去的那個「258M」，其實是 HPC 上的舊版，
是 3,507 個物種；不是我以為的 CrucialX9 正確版，那個是 1,535 個物種。兩個檔案
讀數都差不多 258M，路徑長得也像，我就搞混了。而且關鍵是——正確那份是 6/17 才傳到
Nano4 的，但 v10 我 6/11 就丟出去了，那時候正確的資料根本還不在機器上。

第二個是 reader bug，這個待會兒第四、第六張會細講。

喔對，我還順便發現，連 66.6% 那個 baseline 本身都有問題，是 leakage 高估的。
所以今天的工作就是：換對資料、把 reader 修好、重建一個乾淨的 test set，重測。

## Slide 3 — The Scaling Paradox / 規模的矛盾

這張我把「期待」跟「現實」擺在一起，對比會很清楚。

左邊是當初的期待。Deep learning 的常識就是資料越多越好嘛——v9 用 50M 做到 66.6%，
我預期放大到 258M、也就是 5 倍，至少可以再加個 2 到 5 pp，樂觀一點摸到 70%。

右邊是實際發生的事，全部掉下來。v10 開 weights 的，100K test 只有 40.1%；
v12 warm-start 關 weights 的，45.4%；v13 from scratch 關 weights 的，44.5%。

老師可以看到，三個 258M 的點全部擠在 40 到 45% 這一帶，相對 v9 是整整往下掉 22 pp。
所以這禮拜要解釋的核心矛盾就是一句話：**資料變 5 倍，accuracy 反而被砍掉三分之一。**
這在直覺上完全說不通，所以一定有東西壞了。

## Slide 4 — v10 Collapse → Root Cause / 真正的根因

這張先看 v10 為什麼 collapse，然後講我後來查到的真正原因。

左邊這張圖是 v10 的 val_acc 曲線。它從 epoch 1 到 epoch 12 就一直黏在 21% 上下，
完全爬不動，最後 early stopping。綠色虛線是 v9 的 66.6%，橘色虛線是 v10 在 100K test
的 40.1%。一開始我真的以為這是 class_weights 把 loss 扭曲掉的。

可是 v12、v13 關掉 weights 之後還是只有 45%，所以我就認真去做 code 跟 data 的 audit。
右邊就是我 6/18 確認的真正診斷，總共兩個 bug 疊在一起。

第一個，也是最關鍵的——資料集用錯了。v10、v12、v13 餵的全都是 HPC 那個舊版，
3,507 物種；不是正確的 CrucialX9 1,535 物種。你想，training 是用 3,507 物種的
genus 分佈去學的，可是 test 是 1,535 物種的空間，這兩個 label space 根本對不起來，
模型怎麼可能答對。

第二個是 reader bug，這個是複合傷害。258M 的訓練走的是 train_ddp.py 的 lazy loader，
裡面那個 `_read_seq` 只讀每一筆 record 的第一行。偏偏這個 258M 的 FASTA 是 60 欄
wrap 的——一條 150bp 被拆成 60、60、30 三行——所以模型每一條 read 其實只讀到前面
60bp，等於有 60% 的序列直接被丟掉。那為什麼 v9 沒事？因為 v9 走的是 train.py 的
`_parse_fasta`，它會把多行接回來，而且 50M 那個檔本身就是單行的，所以它讀到完整 150bp。
同一個模型，差別只在讀檔。

第三個我順手抓到的——baseline 也有 leakage。我從 100K test 抽了 300 筆 seq_id 去查，
300 筆全部都在 50M 的 training set 裡面。所以 66.6% 是被高估的。

喔，那有人可能會問「會不會就是 genus 不平衡？」我也排除掉了：v9 是 349:1、v13 是 400:1，
幾乎一樣，不平衡解釋不了 22pp 這麼大的落差。class_weights 只是讓 v10 在本來就很爛的
基礎上又更爛一點，40% 對 45% 而已。

## Slide 5 — Four Experiments, One Conclusion / 四個實驗一個結論

這張我把四個實驗的設定跟結果整個攤開來。

v9，50M balanced、from scratch、不加權、30 個 epoch，做到 66.6%，這是 baseline。

v10，258M，但是是 HPC 舊版那個 3,507 物種、warm-start、有加權，epoch 12 early stop，
40.1%，collapse。

v12，一樣 258M HPC 舊版、warm-start、這次關掉加權，只跑到 3 個 epoch，45.4%。

v13，一樣 258M HPC 舊版、改成 from scratch、也是關加權，完整跑滿 25 個 epoch，44.5%。

下面三個備註。最重要的是：這三個 258M 的實驗，用的全都是 HPC 舊版 3,507 物種，
不是正確的 CrucialX9——這是最主要的根因。另外，v10 是因為 val_acc 不動，
在 epoch 12 就 early stop 了，本來排 15。v12 為什麼只跑 3 個 epoch？因為它分到的
節點 NFS cache 是冷的，一個 epoch 要 476 分鐘，直接撞到 23 小時的 wall time——
這個 infra 的問題第八張會講。

那最重要的結論在底下這個黃框：三個 258M 實驗，不管你 warm-start 還是 from scratch、
不管你加不加權、不管你跑 3 個還是 25 個 epoch，全部收斂在 44 到 45%。
**這種「什麼變因都撼動不了天花板」的現象，就是在告訴你問題出在三個實驗共用的那個
「輸入」，而不是初始化或加權。**這正是為什麼我會跑去 audit 讀檔流程，
最後抓到那個把 150bp 砍成 60bp 的 reader bug。

## Slide 6 — Key Finding: Two Compounding Bugs / 兩個疊加的 bug

這張是這禮拜最重要的 takeaway，而且坦白說，結論跟我一開始猜的完全不一樣。

這 22pp 的落差，根本不是資料不平衡造成的，是兩個 bug 同時存在。

Bug 1，資料集用錯了，這是主因。v10、v12、v13 用的是 HPC 舊版 3,507 物種，
不是正確的 CrucialX9 1,535 物種。兩份資料的物種組成、genus 分佈都不一樣，
那訓練出來的 classifier 拿去測 1,535 物種的 test set，當然對不上。

Bug 2，讀檔截斷，這是複合上去的傷害。就算我給它正確的資料，train_ddp.py 的
`_read_seq` 還是只讀第一行；而這個 FASTA 是 60 欄 wrap，所以每條 read 只讀到
前面 60bp，等於只看到 40% 的序列。v9 那邊用的 `_parse_fasta` 會把多行接起來，
加上 50M 的檔本身單行，所以它沒踩到。

右邊三個結論。

第一，volume 跟 balance 這兩件事，其實都還沒被真正測試過。因為所有 258M 的 run
用的都是錯的資料，在修掉這兩個 bug 之前，我對「資料量有沒有用」「平衡有沒有用」
其實一個結論都下不了。

第二，66.6% 那個 baseline 是漏的。100K test 完全包在 50M training 裡面，
所以這個數字偏高，真正乾淨的 baseline 會比這個低。

第三，好消息是——正確的資料已經到位了。CrucialX9 1,535 物種版，258M、47GB，
6/17 已經傳上 Nano4，reader 修好之後就可以直接跑正確的實驗。

所以一句話總結這禮拜：最大的收穫不是某一個分數，而是抓到兩個讓所有 258M 實驗
同時失真的 bug，而且修正要用的材料都已經備好了。

## Slide 7 — Fix in Action / 修正版正在跑

這張是這次報告最新、我覺得也最關鍵的一頁——修正之後的兩個實驗，**現在這個當下
就正在 Nano4 上跑**，而且結果已經直接驗證了我前面的假說：問題是出在資料，
不是模型、也不是規模。

這兩個 run 用的都是正確的 CrucialX9 1,535 物種資料，而且我 subsample 的時候
輸出成單行 FASTA，所以前面講的兩個 bug 同時被繞過去了——資料是對的，
reader 截斷也不會發生。

先看 v14，就是藍色這條線，job 121437。250M balanced、完全 from scratch。
現在跑到 epoch 7，val_acc 已經到 62.0%，而且還在往上爬、還沒 plateau。
但我想請老師特別看的是——它在**第 2 個 epoch 就已經越過 v10/v12/v13 那條 44.5%
的天花板了**，就是圖上那條紅色虛線。同樣的模型、同樣的規模，我什麼都沒改，
只是把資料換對，它就直接甩開所有舊的 258M 實驗 22 個百分點。這就是「落差來自
資料、不是模型」最直接的證據。

再看 v15，綠色這條，job 122325。同一份資料，但我從 v9 做 warm-start，
然後 learning rate 調低 10 倍，降到 3e-5。它第一個 epoch 就直接停在 66.5%，
而且穩穩的沒有掉。這裡的對照是 v12——v12 當初 warm-start 是從 66% 一路崩到 45%。
這次沒有重演，代表低 LR 這個策略是對的。

不過我要很誠實講一個但書，就是圖右下角那個黃框：這兩個數字都還是
training 過程中的 validation accuracy，而且是在 balanced data 上算的，
還不是最終那個乾淨的測試數字。真正的獨立 100K test，要等 run 收斂之後再跑一次
才算數。但就算把這個但書放進去，趨勢已經非常清楚了：換對資料之後，
連 from scratch 都能輕鬆超過所有舊實驗，warm-start 更是直接守在 v9 的水準。

## Slide 8 — Infrastructure Lessons / 64-GPU DDP 的工程教訓

這張講一下這禮拜在 64-GPU DDP 上學到的工程教訓，這對之後跑大規模訓練蠻重要的。

左邊這張圖是這禮拜最戲劇性的數字。同樣一個 epoch，如果分到的節點 NFS cache 是冷的，
像 v12，要跑 476 分鐘；如果分到 cache 是熱的，像 v13，只要 43 分鐘——差了 11 倍。
而這個差別純粹看你的 job 落在哪一組節點、那組節點有沒有把 47GB 的 FASTA index
快取起來，跟模型本身一點關係都沒有。

右邊是 DDP 的設定跟瓶頸。規模是 8 個節點乘以 8 張 H200，總共 64 張 GPU；
batch 是每張卡 128，乘起來 effective batch 8192；真正的瓶頸是 lazy FASTA index
要走 NFS，258M reads、47GB；冷熱差就是剛剛講的 476 對 43 分鐘。
另外 quota 也要喬——兩個 job 都要 64 張卡會撞到 MaxGRESPerAccount，
所以我 v12 用一個帳號、v13 用另一個帳號分流。然後有 23 小時的 soft wall limit，
我靠 last.pt 自動 resume 接力跑。

底下綠框的教訓就是：大型 DDP run 開跑之前，最好先把 FASTA index 預熱或 pin 住，
不然冷 cache 的 run 等於用 10 倍的 GPU-hours 去跑同一個 epoch，非常浪費資源。

## Slide 9 — Next Steps / 接下來的計畫

最後講接下來怎麼走。核心原則是：在修好之前，所有 258M 的數字都不能信，
所以順序很重要，不能跳。

第一步，把兩個 bug 修掉、重建 baseline。資料切換成正確的 CrucialX9 1,535 物種版，
這個已經在 Nano4 上了。`_read_seq` 要改成讀到下一個 `>` 為止，把 wrap 的多行接起來——
因為 CrucialX9 也是 60 欄 wrap，同一個 bug 會再發生一次——然後在訓練啟動的時候
確認讀進來的長度真的是 150bp。還有，重建一個乾淨的 100K test set，
seq_id 跟 training 完全不重疊，再用它把 v9 重新評估一次，拿到一個誠實的 baseline，
這個數字會比 66.6% 低。

第二步，用正確資料重跑——這個其實已經在跑了，就是上一張的 v14 跟 v15。
v14 from scratch 現在 epoch 7、62% 還在升；v15 warm-start epoch 4、66.5% 穩住。
兩個都用對的資料加單行 FASTA，把兩個 bug 都繞過去了。

第三步，等它們收斂之後做乾淨評估，這時候才能下結論。我會用那個零重疊的乾淨
100K test，幫 v14、v15 各跑一次拿到誠實數字。然後 volume 這個問題，我會再補一個
正確的 258M full-length，修好 reader 之後去跟 250M 的平衡組對照，這樣才能把
「資料量」這個變因單獨隔離出來。

所以這禮拜對我來說，最重要的不是一個新的高分，而是抓到一個會讓所有大規模實驗
失真的 reader bug。我覺得這對論文反而是好材料——一個「naive scaling 因為一個
pipeline bug 而失敗、然後被系統性地找出來並修正」的除錯故事，比直接報一個漂亮數字
要扎實得多。以上，謝謝老師。

## 備用 Q&A

**Q：為什麼一開始要開 class_weights？**

因為 258M 在 genus level 有 400:1 的不平衡，加權本來就是處理不平衡最標準的做法，
所以我 v10 就直接開了。但事後看這是個誤判——關掉之後 v12、v13 一樣只有 45%，
代表加權根本不是主因。真正的問題是讀檔，reads 被截成 60bp，這跟加權、
跟平不平衡都沒關係。class_weights 只是讓 v10 在那個爛基礎上又額外更糟一點而已。

**Q：v12 只跑了 3 個 epoch，這數字還能拿來比嗎？**

可以。v12 是卡在冷 NFS，一個 epoch 476 分鐘，撞 wall time 只跑到 epoch 3。
但它那 3 個 epoch 已經收斂到 45.4%，而且 warm-start 是從 66.3% 一路掉到 45%，
趨勢非常明確。再加上 v13 完整跑滿 25 個 epoch 也是 44.5%，兩個互相佐證，
所以這個結論是穩的。

**Q：258M 跟 50M 都是 species-balanced，為什麼差這麼多？**

主要是三件事同時發生。第一，258M 那個根本是錯的資料集，HPC 舊版 3,507 物種，
不是正確的 1,535 物種，label space 對不起來。第二，reader 只讀第一行，
60 欄 wrap 的檔被截成 60bp，丟掉 60% 的序列。第三，66.6% 本身有 leakage、是高估的。
這三個疊在一起，才會掉這麼多。

**Q：reader 修好之後，如果 258M 還是回不到 baseline 怎麼辦？**

先講清楚，要比的是「乾淨的 v9 baseline」，不是那個被高估的 66.6%。如果修好 reader、
用完整 150bp 重跑之後，258M 還是贏不過乾淨 baseline，那才輪到下一層假設——
genus balance、train/test 分布差異、或 genome 來源不同。那時候 v14 那個平衡的 250M
就是用來隔離「平衡」這個變因的對照組。但這些都要等 reader 修好、有乾淨 test set
之後才有意義，不然就是在錯的數字上面繞圈圈。

**Q：這個負面結果對論文有沒有價值？**

有，而且是好材料——但價值在於 debug 的嚴謹，不在「平衡」這個結論，因為那還沒被證實。
故事是這樣：5× 資料看起來讓 accuracy 崩了，我透過系統性的對照——warm 對 scratch、
加權對不加權、不同 epoch 數——全部收斂在同一個天花板，先排除掉初始化跟加權，
最後用 code+data audit 定位到一個讓所有大規模實驗失真的 reader bug，然後修掉它。
這種「反直覺結果 → 有紀律地排除假設 → 找到真因」的過程，比直接報一個高分
更能展現研究的可信度。
